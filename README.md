# File Server

File Server - веб-файловый сервер на Flask для личного и общего хранения файлов. Приложение дает пользователям личные кабинеты, персональное хранилище, общие папки с правами доступа, админ-панель и браузерный файловый менеджер.

Проект рассчитан на локальное или домашнее использование: можно запустить сервер на машине в локальной сети и открывать файловое хранилище с других устройств через браузер.

## Основные возможности

- авторизация пользователей по логину и паролю;
- личное хранилище для каждого пользователя;
- общие папки с индивидуальным доступом;
- загрузка одного или нескольких файлов;
- загрузка больших файлов чанками;
- RAM-буфер для ускоренной приемки небольших файлов;
- фоновая запись файлов из RAM на диск;
- индикатор RAM-буфера на дэшборде: занятость, скорость записи на диск и ETA;
- создание папок;
- копирование, вырезание и вставка файлов/папок;
- переименование и удаление элементов;
- скачивание файлов;
- админ-просмотр всех файлов сервера;
- управление пользователями;
- выдача ролей и прав доступа;
- смена пароля и аватара;
- светлая, темная и автоматическая тема интерфейса;
- дэшборд с квотой хранилища, профилем и быстрыми переходами.

## Модели данных

В `models.py` описаны основные таблицы:

- `User` - пользователь, хеш пароля, роль администратора и выбранная тема;
- `File` - запись файла для совместимости с ранней логикой загрузки;
- `SharedFolder` - общая папка;
- `SharedFolderAccess` - связь пользователя и общей папки;
- `UserPermission` - дополнительные права пользователя: создание и редактирование общих папок.

## Настройка `.env`

Основные параметры задаются в `.env` в корне проекта:

```env
FILE_SERVER_STORAGE_PATH=/home/admin/file-server/uploads
FILE_SERVER_STORAGE_QUOTA=10GB
FILE_SERVER_UPLOAD_CHUNK_SIZE=8MB
FILE_SERVER_UPLOAD_RAM_BUFFER=1GB
FILE_SERVER_SECRET_KEY=change-me
FILE_SERVER_DATABASE_URI=sqlite:///db.sqlite
```

- `FILE_SERVER_STORAGE_PATH` - папка, где физически лежат загруженные файлы. Можно указать абсолютный путь или путь относительно проекта.
- `FILE_SERVER_STORAGE_QUOTA` - общий лимит хранилища. Поддерживаются значения вроде `500MB`, `10GB`, `1.5TB` или число байт.
- `FILE_SERVER_UPLOAD_CHUNK_SIZE` - размер чанка при загрузке больших файлов.
- `FILE_SERVER_UPLOAD_RAM_BUFFER` - объем оперативной памяти, который можно использовать как буфер загрузки. `0` отключает RAM-буфер.
- `FILE_SERVER_SECRET_KEY` - секретный ключ Flask-сессий. В реальном запуске лучше задать свой длинный случайный ключ.
- `FILE_SERVER_DATABASE_URI` - строка подключения SQLAlchemy. По умолчанию используется SQLite.

После изменения `.env` нужно перезапустить сервер.

## Как устроено хранилище

Файлы лежат внутри директории из `FILE_SERVER_STORAGE_PATH`.

Типовая структура:

```text
uploads/
  personal/
    user_1/
    user_2/
  shared/
    music/
    video/
    photo/
    documents/
    folder-xxxxxxxx/
  avatars/
    user_1.jpg
  .upload_tmp/
    chunks/
```

Личные файлы изолированы по ID пользователя. Общие папки хранятся отдельно и доступны только тем пользователям, которым выданы права.

## Загрузка файлов чанками

Загрузка больших файлов реализована в `static/chunked-upload.js`.

Frontend:

1. Берет выбранные файлы.
2. Делит каждый файл на чанки.
3. Для каждого чанка отправляет `FormData` через `fetch`.
4. Передает имя файла, `upload_id`, номер чанка, общее количество чанков, позицию чанка и полный размер файла.
5. Показывает прогресс, процент и скорость отправки.

Backend принимает чанки в `write_chunk_to_path()` в `app.py`.

Сервер проверяет:

- корректность имени файла;
- порядок чанков;
- общий размер файла;
- квоту хранилища;
- текущую папку назначения;
- доступ пользователя к папке.

Если все чанки пришли корректно, файл сохраняется в нужную директорию.

## RAM-буфер загрузки

RAM-буфер реализован классом `RamUploadBuffer` в `app.py`.

Задача буфера - быстро принять небольшой файл в оперативную память, чтобы клиент быстрее завершил отправку, а запись на диск произошла фоном.

Логика работы:

1. Сервер получает первый чанк файла.
2. Проверяет `FILE_SERVER_UPLOAD_RAM_BUFFER`.
3. Если файл меньше лимита и в буфере есть место, создается RAM-буфер.
4. Чанки добавляются в `bytearray`.
5. После последнего чанка файл попадает в очередь записи.
6. Фоновый поток пишет файл из RAM на диск.
7. Дэшборд показывает текущую занятость RAM, скорость записи и ETA.

Если файл больше RAM-лимита или буфер занят, сервер использует обычный безопасный fallback: сборку файла во временный `.part` файл на диске.

### Статус RAM-буфера

Маршрут:

```text
GET /dashboard/ram-buffer-status
```

Возвращает JSON со статусом:

```json
{
  "enabled": true,
  "limit_bytes": 1073741824,
  "used_bytes": 0,
  "percent": 0,
  "speed_bytes_per_sec": 0,
  "eta_seconds": 0,
  "pending_files": 0,
  "active_file": "",
  "flushing": false
}
```

`static/app.js` опрашивает этот endpoint раз в секунду и обновляет нижнюю полоску на дэшборде.

## Файловые операции

В личном и общем хранилище поддерживаются:

- `create_folder` - создать папку;
- `copy` - скопировать выбранные элементы в session-буфер;
- `cut` - вырезать выбранные элементы в session-буфер;
- `paste` - вставить элементы из буфера;
- `rename` - переименовать один выбранный элемент;
- `delete` - удалить выбранные элементы.

Для защиты от ошибок код не позволяет вставить папку внутрь самой себя или в собственную дочернюю директорию.

## Безопасность

В проекте реализованы базовые меры защиты:

- приватные страницы закрыты через `login_required`;
- админские действия проверяются через `require_admin()`;
- пароли хранятся как хеши Werkzeug;
- имена файлов проходят очистку через `sanitize_entry_name()`;
- относительные пути проходят нормализацию через `sanitize_relative_path()`;
- путь назначения проверяется, чтобы пользователь не вышел за пределы разрешенной директории;
- совпадающие имена файлов получают безопасный суффикс;
- перед загрузкой проверяется квота хранилища.

## Ограничения текущей версии

- `python app.py` запускает встроенный Flask development server. Для постоянного production-запуска лучше использовать Gunicorn/uWSGI, systemd и reverse proxy.
- RAM-буфер хранится в памяти процесса. Если сервер перезапустить до завершения фоновой записи, файл, который еще не записан на диск, может потеряться.
- При запуске с несколькими worker-процессами RAM-буфер будет отдельным у каждого процесса. Для production-версии лучше использовать внешнюю очередь задач.
- Дефолтный пароль администратора из `create_admin.py` нужно менять сразу после первого входа.
- `FILE_SERVER_SECRET_KEY` нужно задавать вручную для реального использования.
